# .NET Debug Adapter

The .NET debug adapter (`@debugmcp/adapter-dotnet`) enables AI agents to debug .NET applications through the MCP protocol. It uses [netcoredbg](https://github.com/Samsung/netcoredbg), Samsung's MIT-licensed .NET debugger.

## Architecture

```
MCP Client → MCP Server → SessionManager → ProxyManager → ProxyWorker
                                                              ↓
                                                    DotnetAdapterPolicy
                                                              ↓
                                                    netcoredbg-bridge (TCP↔stdio)
                                                              ↓
                                                         netcoredbg
                                                              ↓
                                                    .NET Runtime (CoreCLR or Desktop CLR)
```

### Key Components

- **DotnetDebugAdapter** (`src/DotnetDebugAdapter.ts`): Implements `IDebugAdapter`. Handles executable discovery, environment validation, launch/attach config transformation, and PDB conversion.

- **DotnetAdapterFactory** (`src/DotnetAdapterFactory.ts`): Implements `IAdapterFactory` for dependency injection. Creates adapter instances and provides metadata.

- **DotnetAdapterPolicy** (`packages/shared/src/interfaces/adapter-policy-dotnet.ts`): DAP proxy policy encoding netcoredbg-specific behaviors — DAP sequence, scope naming, variable filtering, stack frame filtering, and spawn configuration.

- **netcoredbg-bridge** (`src/utils/netcoredbg-bridge.ts`): TCP-to-stdio bridge that works around a netcoredbg `--server=PORT` bug on Windows. Listens on a TCP port, spawns netcoredbg in stdio mode, and forwards DAP messages bidirectionally.

- **dotnet-utils** (`src/utils/dotnet-utils.ts`): Utility functions for netcoredbg discovery, PDB format detection, and Windows PDB to Portable PDB conversion via Pdb2Pdb.

### Why a TCP-to-stdio Bridge?

netcoredbg supports DAP over TCP via `--server=PORT --interpreter=vscode`, but has a connection bug on Windows where the TCP connection drops after the DAP initialize sequence. The bridge spawns netcoredbg in stdio mode (which works reliably) and exposes a TCP socket for the proxy to connect to. This is a pure byte-level forwarder with no DAP parsing.

## Supported Runtimes

### .NET Core / .NET 5+ (Stock netcoredbg)

Works out of the box with the standard Samsung netcoredbg release. No special configuration needed.

- Install netcoredbg from [releases](https://github.com/Samsung/netcoredbg/releases) or build from source
- Portable PDB symbols are generated by default

### .NET Framework 4.x (Forked netcoredbg)

Requires a [modified netcoredbg fork](https://github.com/bob7123/netcoredbg/tree/framework-support) that adds Desktop CLR support via ICorDebug COM interfaces.

**Why a fork?** Stock netcoredbg only supports CoreCLR (.NET Core / .NET 5+). Both runtimes use the same ICorDebug debugging interfaces, but runtime discovery and initialization differ. The fork adds ~240 lines of `#ifdef WIN32` code for Desktop CLR discovery via `mscoree.dll` / `ICLRMetaHost`.

**Setup:**
1. Build the forked netcoredbg from the `framework-support` branch
2. Set `NETCOREDBG_PATH` to point to the built binary

**PDB format requirement:** .NET Framework compilers produce Windows PDB format by default. netcoredbg only reads Portable PDB format. Options:
- Compile with `/debug:portable` (Roslyn csc)
- Use the bundled `Pdb2Pdb.exe` tool (`tools/pdb2pdb/`) which the adapter runs automatically during attach

## Configuration

### Environment Variables

| Variable | Description |
|----------|-------------|
| `NETCOREDBG_PATH` | Path to netcoredbg executable. Checked first during discovery. |
| `PDB2PDB_PATH` | Path to Pdb2Pdb.exe for Windows PDB conversion. Falls back to bundled version. |

### Executable Discovery Order

1. `NETCOREDBG_PATH` environment variable
2. Preferred path (if provided by caller)
3. `which netcoredbg` (searches PATH)
4. Common install locations (platform-specific)

## DAP Protocol Details

- **Adapter ID:** `coreclr` (for both .NET Core and .NET Framework)
- **DAP sequence:** Standard — `initialize → initialized event → attach/launch → configurationDone`
- **Handshake:** No strict handshake required (`requiresStrictHandshake: false`)
- **Local scope name:** `Locals`
- **Safety:** `terminateDebuggee` is always `false` when detaching from attached processes

### Supported Features

- Conditional breakpoints
- Function breakpoints
- Exception breakpoints (`all`, `user-unhandled`)
- Evaluate for hovers
- Set variable
- Exception info
- Loaded sources
- Module loading

### Filtered Variables

C# compiler-generated variables are filtered out by default:
- `<>c__DisplayClass*` (closure classes)
- `CS$<>*` (compiler temporaries)
- `$VB$*` (VB.NET compiler variables)
- `<>t__*`, `<>s__*` (async state machine fields)

### Filtered Stack Frames

By default, stack frames are filtered to show only user code:
- Frames with no source file are hidden
- `System.*` and `Microsoft.*` runtime frames are hidden
